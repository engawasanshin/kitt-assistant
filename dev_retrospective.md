## 2. 今回のデバッグで機能した「基本動作」

### 2-1. 問題を一つずつ切り分ける

失敗パターン（やりがちだが効果が薄い）：
- 「全部まとめて直して」→ 何が直ったか、何が壊れたかわからなくなる
- エラーメッセージを見ずに「なんとなく直す」→ 本質を外す

**成功したアプローチ：**
1. エラーメッセージを正確に読む（「Unterminated string at position 463」→ JSONが途中で切れている）
2. 原因を一つに絞り込む（Geminiのトークン上限が原因 → `maxOutputTokens: 2048 → 4096`）
3. 修正して動作確認してから次へ

### 2-2. 「できている / できていない」をリスト化する

KITTとの会話でたどり着いたモジュール別課題リストが、デバッグを加速させた。

```
A. コア機能（記憶・同期）
B. API連携（データ取得）
C. ウォッチリストUI
D. チャート・分析
E. 自己診断
```

**各モジュールについて「あるべき姿 / 今の姿 / 差分」を明確にする。**
差分が修正タスクになる。これをやらないと「なんか動かない」が延々続く。

### 2-3. コンソールログを武器にする

ブラウザの開発者ツール（F12）のコンソールに流れるログが唯一の「内部の真実」。
今回追加した `[stooq] 成功: query2.allorigins` などのログが、
どのプロキシが機能したかを即座に教えてくれた。

**次回以降の原則：重要な処理には必ず `console.log('[モジュール名] 処理内容')` を入れる。**

## 3. 技術パターン：今後のアプリに適用できる設計

### 3-1. 外部API取得は「多段フォールバック」が必須

今回の最大の学び。単一のAPIに依存すると必ず壊れる。

```
[優先順1] 直接アクセス（最速・制限なし）
    ↓ 失敗したら
[優先順2] CORSプロキシA（allorigins.win）
    ↓ 失敗したら
[優先順3] CORSプロキシB（corsproxy.io）
    ↓ 失敗したら
[優先順4〜] 別プロキシ / 別APIソース
```

**実装パターン（JavaScript）：**
```javascript
const sources = [
  () => fetch(directUrl),
  () => fetch(`https://api.allorigins.win/raw?url=${encodeURIComponent(directUrl)}`),
  () => fetch(`https://corsproxy.io/?${encodeURIComponent(directUrl)}`),
];
for (const tryFetch of sources) {
  try {
    const res = await tryFetch();
    if (res.ok) { /* 成功 */ break; }
  } catch (e) { continue; }
}
```

**適用場面：** 株価API、天気API、ニュースAPIなど外部データを使うすべての機能

### 3-2. APIコール回数を最小化する（トークン節約・レート制限対策）

今回の失敗：Alpha Vantageの無料枠（25回/日）をデバッグ中の再試行で消費してしまった。

**原則：**
- 取得したデータは**必ずキャッシュ**（localStorage / IndexedDB）する
- キャッシュが有効な間はAPIを叩かない（有効期限: 15分〜1時間程度）
- バックグラウンドでの自動更新は**明示的に制御**する（画面を開くたびに叩かない）

```javascript
const cache = JSON.parse(localStorage.getItem('data_cache') || '{}');
const now = Date.now();
if (cache[key] && now - cache[key].fetchedAt < 15 * 60 * 1000) {
  return cache[key].data; // キャッシュを使う
}
// キャッシュなし or 期限切れ → APIを叩く
```

### 3-3. GeminiのJSON出力は「トークン上限」に注意

今回発生した「Unterminated string in JSON at position 463」の根本原因。

**設定の鉄板：**
```javascript
generationConfig: {
  temperature: 0.3,
  maxOutputTokens: 4096,    // 2048は日本語JSONでは足りない
  responseMimeType: 'application/json',  // JSON強制
}
```

**仕上げの安全網：**
```javascript
if (candidate?.finishReason === 'MAX_TOKENS') {
  throw new Error('Geminiが途中で止まりました。再試行してください。');
}
try {
  analysis = JSON.parse(rawText);
} catch (e) {
  console.error('[gemini] rawText先頭:', rawText.slice(0, 200));
  throw new Error('JSON解析失敗: ' + e.message);
}
```

### 3-4. AIの会話と「実データ」を分離する

今回の重要な設計判断。KITTの会話AIが「架空の株価」を生成してしまう問題。

**原則：AIが「知っていること」と「アプリが持っているデータ」は別物。**

- 会話AI（Gemini）→ コミュニケーション・判断・提案を担当
- アプリのロジック → 実データ取得・計算・表示を担当
- 両者が混ざると「それっぽい嘘」が生まれる

**実装：** システムプロンプトに明示的な禁止事項を書く
```
【絶対禁止】
- 株価数値を生成すること
- 架空のチャートや分析を述べること
→ 正しいデータはアプリのウォッチリスト経由のみ
```

### 3-5. バージョン番号でデプロイを追跡する

「修正したはずなのに変わってない」問題の解決策。

```
v2026-03-30a → v2026-03-30b → ... → v2026-03-30f
```

画面上に表示することで「どのバージョンが動いているか」がすぐわかる。
GitHub Pagesのキャッシュ問題も `Cache-Control: no-cache` メタタグで対処済み。

## 4. Claude Codeとの協業で効率を上げるコツ

### 4-1. 事前調査を先にやる（仕様確認ファースト）

今回後から発覚したこと：
- Alpha Vantageの無料枠は**25回/日**（最初500と思っていた）
- stooq.comは**CORSを許可していない**（プロキシ必須）
- Geminiの無料枠は**`maxOutputTokens`のデフォルトが低い**

**次回から：** 外部APIを使う機能を実装する前に、以下を先に確認する
1. 無料枠の制限（回数/日、回数/分）
2. CORSポリシー（ブラウザからの直接アクセス可否）
3. レスポンス形式（JSON/CSV、フィールド名、データ型）
4. 日本語・日本株への対応状況

### 4-2. 「動いた証拠」を確認してから次へ

「修正しました → 次の修正へ」ではなく、
**「修正しました → 動作確認しました → 次へ」** のサイクルを守る。

確認方法：
- コンソールに成功ログが出ているか
- 画面上で期待通りの動作をしているか
- iPhoneとMac両方で確認（環境差異がある）

### 4-3. チャット分割でトークンを節約する

Claude Codeの制限：5時間ローリングウィンドウ（約10〜40プロンプト）

**効率化：**
- タスクが変わったら新しいチャットを開く
- `MEMORY.md` と `.ai_context/status.md` に現状を書いておく → 次のチャットで即再開できる
- 大きな修正は「計画チャット → 実装チャット」に分ける

### 4-4. エラーはそのままコピペする

「なんかエラーが出た」ではなく、コンソールのエラーをそのままコピペすると、
Claude Codeが原因を正確に特定できる。

## 5. 今回のGemini分析の品質について

完成したSCREEN・INPEXの分析は実用レベルに達している：

```
【トレンド判定】下降トレンド
現在の株価は5日MAと25日MAの両方を大きく下回っています。
5日MAが25日MAを下回っており、下降トレンドの勢いが強いことを示唆しています。

【ボリンジャーバンド】
現在の株価(9119.0)は下限(8931.4)に非常に近い位置にあります。
これは短期的に売られすぎの状態にある可能性を示唆しています。

【注目ポイント】
1. BB下限付近での株価の動き：反発するのか、さらに下値を試すのか
2. 反転の兆候：ハンマー足など反転シグナルが出たら
```

**次のアプリへの適用可能性：**
- 財務データ分析（売上トレンド、コスト推移）
- ヘルスデータ分析（睡眠、運動パターン）
- 家計分析（支出トレンド、カテゴリ別傾向）

構造は同じ：**「数値データ → Geminiに渡す → 構造化されたJSONで分析返却 → 画面表示」**
